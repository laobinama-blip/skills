# Westwell-Cosmo Team SOP (One Page)

## 目标

用统一流程把 AI 开发变成可控交付：

`需求 -> 探索 -> 规格 -> 实现 -> 硬化 -> 归档 -> PR -> 发布`

## 四类角色

- 业务/产品：给出目标、边界、验收标准。
- 开发：按流程执行，不跳阶段。
- Reviewer：审核规格一致性与质量门禁。
- 平台/架构：维护全局规则、模板、度量。

## 需求分级

- 小改动：文案、日志、微重构。
- 常规需求：新增接口、流程调整。
- 复杂重构：拆模块、改鉴权、换库、架构迁移。

## 标准流程（五阶段）

前置条件（进入流程前）：

- 必须有完整需求包：目标、范围、非目标、约束、验收标准、边界条件。
- 需求不完整时，不进入设计与实现，只做需求补齐。
- 每个阶段的每个输出物都必须确认通过，未确认不得进入下一阶段。

输出物确认范围（文档与附件一并确认）：

- 文档：PRD / AC / SPEC / TestCases / SecurityReport / ReleasePlan / Postmortem
- 表格：Traceability 矩阵（如 xlsx）
- 证据链接：CI 流水线链接、测试报告链接、安全扫描链接
- 附件：架构图、流程图、截图、录屏、PDF、原型图等

1. 探索阶段（Superpowers）
- 先 `brainstorm`，再 `plan`，禁止直接写代码。
- 输出：目标/边界/非目标、方案对比、风险与测试思路。
- Gate-1：Phase 1 输出物逐项确认后，才能进入规格阶段。

2. 规格阶段（OpenSpec / SDD）
- 产物：`proposal.md`、`design.md`、`tasks.md`、`delta spec`（行为变更必需）。
- Gate-2：文档齐全并逐项确认通过，才能进入编码阶段。

3. 实现阶段（Superpowers + TDD）
- 先测后码（RED -> GREEN -> REFACTOR）。
- 小步提交，任务逐项完成并验证。
- PR 必须关联 OpenSpec 路径。
- Gate-3：实现产物逐项确认通过，且代码、测试、规格一致后进入硬化阶段。

4. 硬化阶段（Hardness Gate）
- Security：SAST/SCA 通过，高危问题清零或有批准例外。
- Stability：单测、集成测试通过。
- Observability：日志/指标/链路齐备。
- Rollback：灰度策略和回滚方案明确。
- Compatibility/Perf：兼容性和性能回归可接受。
- Gate-4：硬化产物逐项确认通过后进入归档。

5. 归档阶段（OpenSpec）
- 先 verify，再 archive。
- 条件：任务全完成、验证结论明确、与规格一致。

## 绝对红线

- 未确认设计，不进规格阶段。
- 未补齐规格，不进编码阶段。
- 无真实测试结果，不宣称完成。
- 代码/测试/规格不一致，不允许归档。

## 合并门禁（最低）

- Lint 通过
- 单元测试通过
- 集成测试通过
- 覆盖率 >= 80%（核心模块建议 >= 90%）
- 构建成功
- 至少 1 名非提交者评审通过

## KPI（建议）

- AC 达成率
- 一次提测通过率
- CI 通过率与平均修复时长
- 线上缺陷率与变更失败率
- MTTR
- 有 spec 的需求占比
- verify 覆盖率

## 沟通话术（统一口径）

- “先走探索，不直接改代码。”
- “先补 OpenSpec，再做实现。”
- “先过硬化闸门，再提归档。”
- “没有 verify 结论，不合并。”
